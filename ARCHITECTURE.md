# Архітектура Проєкту "Trade Bot V3"

Цей документ описує загальну архітектуру торгового бота, взаємодію його компонентів та принцип роботи під час розгортання в Docker/Coolify середовищі.

## 1. Загальна Структура

Проєкт побудований за класичною клієнт-серверною архітектурою, розбитий на два основні сервіси:

- **Backend (Python / FastAPI):** Відповідає за алгоритмічну торгівлю, зв'язок з Binance, розрахунок рівнів (сіток) на основі мульти-таймфрейм аналізу та управління ордерами.
- **Frontend (React / Vite / TypeScript):** Відповідає за зручний інтерфейс користувача (Dashboard), налаштування параметрів торговельної стратегії в реальному часі та відображення логів/статусу.

---

## 2. Backend (Бекенд)

Бекенд — це ядро торгової логіки. Він працює асинхронно та використовує вебсокети для отримання актуальних ринкових даних без затримок.

### Основні модулі:
1. **`server.py` (FastAPI Server):**
   - Точка входу в застосунок.
   - Надає REST API (наприклад, `/api/calculate_grid`, `/api/clear_grids`) для ручного управління.
   - Тримає WebSocket-з'єднання з Frontend-ом (`/ws/stream`) для передачі логів, статусу бота та оновлення сітки в режимі реального часу.
   - Зберігає глобальний стан (`bot_ctx`), який містить вибрану торгову пару та поточні параметри.

2. **`main_live_trading_manager.py` (LiveTradingManager):**
   - Серце торгового бота. Керує життєвим циклом торгівлі: перевіряє умови входу, слідкує за відкритими позиціями, ініціює перерахунок сіток та закриття по Take Profit (TP) / Stop Loss (SL).
   - Запускається у фоновому циклі, постійно аналізуючи `bot_ctx`.

3. **`market_analyzer_module.py` (MarketAnalyzer):**
   - Виконує мульти-таймфрейм аналіз (наприклад, 1m, 5m, 15m, 1h, 4h, 1d).
   - Завантажує свічки (Klines) з Binance.
   - Розраховує технічні індикатори: смуги Боллінджера (Bollinger Bands), ковзні середні (SMA, EMA), рівні Фібоначчі (Fibonacci Retracement), локальні екстремуми та Ішімоку.

4. **`grid_generator_module.py` (GridGenerator):**
   - На основі даних від `MarketAnalyzer` формує динамічну сітку ордерів.
   - Визначає "кластери" сили рівнів (чим більше індикаторів збігається в одному ціновому діапазоні, тим сильніший рівень і тим більша маржа виділяється на цей кластер).

5. **`binance_price_streamer.py` / `binance_live_handler.py`:**
   - Зв'язок із біржею Binance.
   - Price Streamer: Використовує Binance WebSocket API для отримання ціни в реальному часі (зменшує ризик API банів порівняно з REST-опитуваннями).
   - API Handler: Виконує розміщення ордерів, скасування та отримання балансів.

## 3. Frontend (Фронтенд)

Інтерфейс користувача створений для зручного та швидкого налаштування стратегії.

- **Стек:** React, Vite, TypeScript.
- **Взаємодія:** 
  - На старті (або при виборі валюти) робить запит на бекенд і підтягує останні налаштування для конкретної монети.
  - Зберігає постійне WebSocket-з'єднання.
  - Усі події (оновлення сітки, спрацювання ордера, помилки біржі) миттєво з'являються в панелі "Логи" та у таблиці "Orders".

---

## 4. Docker & Інфраструктура (Coolify)

Проєкт підготовлено до розгортання в середовищі Docker за допомогою файлу `docker-compose.yml`.

### Архітектура розгортання:
- **`backend` сервіс:** Збирається з `backend/Dockerfile` (на базі `python:3.11-slim`). Відкриває порт `8000`. Змонтовані томи (volumes) зберігають налаштування `bot_config_cache.json` та API ключі між перезапусками контейнера.
- **`frontend` сервіс:** Збирається в два етапи (multi-stage build). Спочатку збирається статика через Node.js, потім статика "віддається" через легковаговий сервер Nginx. Відкриває порт `80`.

### Зв'язок у контейнерах:
Фронтенд звертається до бекенду через проксі-запити (або вказавши прямий домен вашого API після деплою в Coolify). У конфігурації Docker Compose сервіси знаходяться в одній мережі.

## 5. Безпека та Rate Limits (Ліміти API)
1. Усі конфіденційні дані (API ключі) зберігаються в ігнорованих Git файлах (`binance_api_keys.json`, `.env`). На сервері вони мають створюватися вручну або передаватися як змінні середовища.
2. Бот мінімізує REST API запити до Binance: ціни слухаються через вебсокети, щоб уникнути блокування за перевищення "Weight" лімітів біржі (1200 в хвилину).
